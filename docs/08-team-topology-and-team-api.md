# Team Topology And Team API

## Core question

What team model is needed for platform reuse to become adoptable, supportable, and coherent?

## Team roles

Use platform owner/team, stream-aligned product team, enabling role, and shared stewardship as responsibility models. Do not assign all platform work to one busy architect by default.

## Interaction modes

Use collaboration when platform shape is still being discovered, facilitating when consumers need help adopting, and X-as-a-Service when the platform is mature enough for self-service consumption.

## Team API fields

Define what is offered, what is not offered, adoption path, docs, examples, support channel, versioning policy, contribution rules, exception process, escalation path, roadmap visibility, feedback loop, and interaction mode.

## Adoption stages

Discovery, first adoption, supported use, local adaptation, mature consumption, and deprecation each need different platform behavior.

## Cognitive-load check

Before assigning platform responsibility to a product team, check product/domain load, learning cost, enablement need, runtime ownership, support, funding, and exceptions.

## Anti-patterns

Self-service by README, stream team owns platform by accident, enabling without capacity, shared ownership without responsibility rows, and missing team API.

## What this must not imply

- This is not a universal methodology.
- This is not proof that one platform shape always wins.
- This does not replace local architecture judgment.
- This should be adapted, not adopted blindly.
