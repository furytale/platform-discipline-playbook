# Platform Productization For Outsourcing

## Core question

When can reusable code from one delivery context become a productized platform asset?

## Reusable code vs productized platform asset

Reusable code helps in one context. A productized platform asset can be adopted by consumers who did not write it and has packaging, docs, tests, versioning, compatibility, support, funding, owner, and deprecation path.

## Asset classification

Classify candidates as product-local code, internal accelerator, shared package/library, product-integrated core, platform asset, or productized service. Do not promote by optimism.

## Productization checklist

Remove or name local assumptions. Define generic capability, supported consumers, packaging, versioning, docs, examples, conformance, support, compatibility, funding, owner, adoption, exceptions, and deprecation.

## Support and adoption model

Choose guided, paired, self-service, or not-ready adoption. Self-service is a maturity state, not a README file.

## Migration and compatibility

Every productized asset needs first-adoption path, breaking-change rule, compatibility promise, deprecation policy, and migration/support owner.

## Anti-patterns

Old project code as platform, examples from real clients, copy-paste accelerator advertised as platform, and one-client-funds-everyone-uses-it are all failure modes.

## What this must not imply

- This is not a universal methodology.
- This is not proof that one platform shape always wins.
- This does not replace local architecture judgment.
- This should be adapted, not adopted blindly.
